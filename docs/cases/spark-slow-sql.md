# 案例：Spark SQL 运行缓慢排查

## 1. 现象

某个离线指标任务运行时间从 20 分钟增加到 2 小时，Spark UI 中最后几个 Task 长时间不结束。

## 2. 初步判断

优先怀疑执行计划变化、输入数据量增长、Shuffle 过大或数据倾斜。

## 3. 排查过程

先查看 SQL 执行计划，确认是否新增大表 Join。再查看 Stage 耗时，发现某个 Shuffle Stage 的少数 Task Shuffle Read 远高于平均值。

## 4. 根因分析

Join Key 中存在大量默认值，导致这些记录在 Shuffle 后集中到少数分区，引发 Task 长尾。

## 5. 解决方案

过滤无效默认值，对保留的热点 Key 单独处理，并对小维表启用广播 Join。

## 6. 结果验证

优化后 Stage Task 耗时分布趋于均匀，任务运行时间恢复到 25 分钟以内。

## 7. 经验沉淀

慢 SQL 排查应先看执行计划和 Stage 指标，不要直接改资源参数。

## 总结输出

Spark SQL 慢通常能落到扫描量、执行计划、Shuffle、倾斜或资源问题上，排查要用指标定位证据。
