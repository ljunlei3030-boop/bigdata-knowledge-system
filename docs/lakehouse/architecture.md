# 整体架构

## 1. 它是什么？

整体架构描述数据平台从数据源到数据服务的端到端链路。

## 2. 它解决什么问题？

它帮助明确每个组件的职责，避免 Kafka、Flink、Paimon、Spark 和 ClickHouse 之间边界混乱。

## 3. 它在整个流程中的位置？

它是湖仓专题的全局视角。

## 4. 底层原理是什么？

数据源进入采集层，写入湖存储，计算层完成加工，服务层承接查询，治理层横向贯穿。

## 5. 典型使用场景

平台建设规划、链路改造、任务排查、成本治理。

## 6. 常见问题

重复建设链路、数据口径不一致、查询引擎承载过多加工职责。

## 7. 优化方案

用分层模型和组件职责边界约束数据流向。

## 8. 和其他技术的区别

架构关注组件协作，不等同于某个单点技术。

## 9. 关联知识

- [Kafka + Flink + Paimon + Spark + ByteHouse](/lakehouse/kafka-flink-paimon-spark-bytehouse)

## 总结输出

整体架构的价值是让每个组件在正确的位置解决正确的问题。
