# 数据治理怎么落地推进？

## 结论先行

数据治理不是一次性治理所有表，而是优先治理核心业务链路。推进顺序可以概括为：

<div class="flow-strip">
  <span>明确目标</span>
  <span>资产盘点</span>
  <span>血缘梳理</span>
  <span>标准制定</span>
  <span>质量规则</span>
  <span>优先级治理</span>
  <span>监控告警</span>
  <span>问题闭环</span>
  <span>持续运营</span>
</div>

治理的目标是让数据做到：

| 目标 | 含义 |
| --- | --- |
| 可找 | 知道有哪些表、字段、指标和负责人 |
| 可信 | 数据完整、准确、一致、及时 |
| 可用 | 下游能稳定消费，不需要反复猜口径 |
| 可管 | 权限、生命周期、成本和质量规则可控 |
| 可追溯 | 能通过血缘定位来源、加工过程和影响范围 |

## 1. 先明确治理目标和范围

治理不要从“全库所有表”开始，而是先选择业务影响大、下游依赖多、访问频率高、经常出问题的核心链路。

优先级通常是：

```text
订单链路
支付链路
用户链路
商品链路
埋点行为链路
经营看板链路
```

这些链路一旦异常，通常会直接影响经营分析、核心看板、业务决策和对外数据服务。

## 2. 盘点数据资产

资产盘点解决的是“平台里到底有什么数据、谁负责、谁在用”的问题。

核心表需要维护以下信息：

| 维度 | 说明 |
| --- | --- |
| 表名 | 表的物理名称和中文含义 |
| 所属层级 | ODS、DWD、DIM、DWS、ADS |
| 主题域 | 交易、用户、商品、营销、流量等 |
| 数据来源 | MySQL、Kafka、Flink CDC、日志、第三方系统 |
| 负责人 | 表负责人、任务负责人、业务负责人 |
| 下游依赖 | 被哪些表、任务、报表、接口使用 |
| 访问频率 | 是否高频查询、是否核心看板依赖 |
| 存储成本 | 数据量、保留周期、是否低价值 |
| 质量规则 | 是否有分区、唯一性、空值、波动校验 |

## 3. 梳理元数据和血缘

治理不能只看单张表，要知道数据从哪里来、经过哪些加工、流向哪些下游。

典型链路如下：

<div class="flow-strip">
  <span>源系统 / Kafka / Flink CDC</span>
  <span>ODS 原始层</span>
  <span>DWD 明细层</span>
  <span>DIM / DWS</span>
  <span>ADS 应用层</span>
  <span>BI / 报表 / 数据服务</span>
</div>

如果经营看板 GMV 异常下降，问题可能不在 ADS 表本身，而在更上游：

```text
Flink CDC 延迟
ODS 分区缺失
DWD 清洗条件误过滤
订单状态枚举变化
DWS 汇总任务失败
ADS 使用了错误分区
```

## 4. 制定数据标准

数据标准解决的是“同一个东西不要有多种叫法、口径和类型”的问题。

常见标准包括：

| 类型 | 示例 |
| --- | --- |
| 表命名规范 | `ods_`、`dwd_`、`dim_`、`dws_`、`ads_` |
| 字段命名规范 | `user_id`、`order_id`、`pay_amount`、`create_time` |
| 数据类型规范 | 金额、时间、ID、枚举值统一类型 |
| 分区规范 | 统一使用 `dt` 或明确业务日期字段 |
| 指标口径规范 | GMV、订单数、支付用户数、退款金额 |
| 数仓分层规范 | 每层做什么、不做什么、谁来复用 |

例如字段统一：

```text
用户 ID：user_id
订单 ID：order_id
商品 ID：product_id
支付金额：pay_amount
创建时间：create_time
更新时间：update_time
分区日期：dt
```

## 5. 建立质量规则

质量规则要覆盖表级、字段级和指标级。

| 规则类型 | 检查内容 |
| --- | --- |
| 表级规则 | 分区是否存在、数据量是否为 0、是否按时产出 |
| 字段级规则 | 主键是否唯一、字段是否为空、枚举是否合法 |
| 指标级规则 | GMV 是否异常波动、DWD 和 DWS 是否一致 |

常见检查 SQL：

```sql
-- 检查 ODS 分区是否有数据
SELECT count(*)
FROM ods_order
WHERE dt = '${bizdate}';
```

```sql
-- 检查 DWD 核心字段和金额
SELECT
    count(*) AS cnt,
    sum(pay_amount) AS pay_amount
FROM dwd_order_detail
WHERE dt = '${bizdate}';
```

```sql
-- 检查主键重复
SELECT
    order_id,
    count(*) AS cnt
FROM dwd_order_detail
WHERE dt = '${bizdate}'
GROUP BY order_id
HAVING count(*) > 1;
```

```sql
-- 检查金额异常
SELECT count(*)
FROM dwd_order_detail
WHERE dt = '${bizdate}'
  AND pay_amount < 0;
```

## 6. 按链路排查治理问题

排查数据治理问题时，不要只盯最终报表，要沿血缘逐层定位。

<div class="flow-strip">
  <span>ADS 看板异常</span>
  <span>DWS 汇总是否正确</span>
  <span>DWD 明细是否完整</span>
  <span>ODS 分区是否产出</span>
  <span>源系统 / CDC / Kafka 是否正常</span>
</div>

排查顺序：

1. 看 ADS 应用层是否为空、延迟或异常波动。
2. 根据血缘找到上游 DWS、DWD、ODS。
3. 做层级间对账，比较明细和汇总结果。
4. 查字段级质量问题，例如主键重复、金额异常、核心字段为空。
5. 查任务调度和上游同步，例如 CDC 延迟、Kafka 堆积、DolphinScheduler 依赖失败。

层级间对账示例：

```sql
SELECT sum(pay_amount)
FROM dwd_order_detail
WHERE dt = '${bizdate}';

SELECT sum(pay_amount_1d)
FROM dws_trade_1d
WHERE dt = '${bizdate}';
```

## 7. 建立监控告警和问题闭环

严重异常可以阻断任务，一般异常做告警。关键不是“发现问题”本身，而是发现后能定位、修复、重跑、通知和复盘。

<div class="flow-strip">
  <span>质量规则发现异常</span>
  <span>告警通知负责人</span>
  <span>根据血缘定位上游原因</span>
  <span>修复任务或数据</span>
  <span>重跑受影响分区</span>
  <span>通知下游影响范围</span>
  <span>复盘并补充规则</span>
</div>

可监控的指标包括：

| 指标 | 含义 |
| --- | --- |
| 核心表质量规则覆盖率 | 核心链路是否都有规则保护 |
| 核心任务准时率 | 核心任务是否按 SLA 产出 |
| 异常平均修复时长 | 从发现到修复需要多久 |
| 无主表数量 | 有多少表没人负责 |
| 长期未访问表数量 | 是否存在低价值数据资产 |
| 存储成本下降比例 | 生命周期和低价值表治理效果 |
| 敏感字段脱敏覆盖率 | 安全治理覆盖情况 |

## 8. 方案代价

数据治理不是没有成本。

| 代价 | 说明 |
| --- | --- |
| 盘点成本高 | 需要梳理大量表、字段、任务和负责人 |
| 规则维护成本 | 业务变化后质量规则和指标口径也要更新 |
| 强阻断风险 | 阻断规则可能影响下游任务，需要谨慎设置 |
| 平台依赖 | 血缘、元数据、质量监控需要平台能力支持 |
| 口径协同成本 | 指标统一需要和业务方反复确认 |
| 生命周期风险 | 历史数据下线需要评估影响范围 |

更合理的推进顺序是：

```text
先治理核心链路
再治理公共层表
再治理应用层表
最后治理低频表和成本问题
```

## 总结输出

数据治理应该按照“明确目标 → 资产盘点 → 血缘梳理 → 标准制定 → 质量规则 → 优先级治理 → 监控告警 → 问题闭环 → 持续运营”的步骤推进。

治理重点不是一次性把所有表都改完，而是先保障核心业务链路稳定可信。通过元数据和血缘知道数据从哪里来、流向哪里；通过标准统一命名、类型、枚举和指标口径；通过质量规则发现分区为空、主键重复、金额异常、指标波动等问题；通过调度告警和问题闭环保证异常能被及时发现、定位、修复和复盘。
